## 1. 文档目的

本文档用于明确 InvoiceBox V6.3 的产品范围、业务流程、页面结构、数据模型、接口需求、异常处理及验收标准，作为产品确认、开发实现和测试验收的共同依据。

## 2. 产品定位

InvoiceBox 是一款完全在用户电脑本地运行的发票整理与报销材料生成工具。

产品帮助用户完成：

1. 批量导入 PDF、OFD 发票；
2. 自动识别发票类型、金额、日期、路线等信息；
3. 快速核对识别结果和异常发票；
4. 将发票整理到不同报销批次；
5. 生成报销单、发票拼版 PDF 和发票明细；
6. 对历史发票进行本地归档、搜索和备份。

### 2.1 核心价值

将原本分散、重复且容易出错的发票整理工作，变成一条清晰的本地工作流：

**导入 → 识别 → 核对 → 归入批次 → 生成报销材料 → 标记完成 → 本地归档**

### 2.2 产品原则

- 完全本地运行；
- 不要求登录或注册；
- 不上传发票、识别结果和使用数据；
- 无网络环境下全部核心功能可正常使用；
- 优先保证数据安全、识别可核对和导出结果准确；
- 保持单机、轻量、下载即用；
- 保持 JSON 文件存储，不引入数据库服务；
- 保持现有 Flask、Vue 3、Element Plus 和 PyInstaller 技术架构。

## 3. 版本背景

InvoiceBox V6.2 已具备以下能力：

- PDF、OFD 批量导入；
- MD5 文件去重；
- OFD 转 PDF；
- 发票 OCR 文本提取；
- 多证据加权分类；
- 金额、日期、路线、销售方等字段提取；
- 识别置信度和“需确认”标记；
- 用户分类纠错学习；
- 分类筛选、关键词搜索和拖拽排序；
- 发票拼版预览；
- 专票单页、普通票双拼；
- 专票打印两份；
- 报销单生成；
- PDF 下载和打印；
- 自定义分类和本地设置；
- 数据备份、恢复相关后端基础函数。

当前主要问题不是缺少识别能力，而是缺少完整的管理和报销闭环：

- 用户无法集中查看所有待确认发票；
- 结构化识别结果没有完整展示；
- 发票无法按报销事项分组管理；
- 缺少批量确认和批量调整能力；
- 缺少可交付的完整报销材料包；
- 备份和恢复能力没有前端入口；
- 已有排序、识别依据、风险点等能力没有充分暴露给用户。

## 4. V6.3 版本目标

### 4.1 核心目标

V6.3 将 InvoiceBox 从“发票拼版打印工具”升级为“本地发票管理与报销整理工具”。

本版本必须完成以下闭环：

1. 用户能够快速发现需要核对的发票；
2. 用户能够查看并修改完整发票信息；
3. 用户能够创建报销批次并整理发票；
4. 用户能够批量处理发票；
5. 用户能够按批次生成完整报销材料；
6. 用户能够备份和恢复全部本地数据。

### 4.2 成功标准

- 新用户不阅读说明，也能完成一次“导入—核对—导出”流程；
- 20 张发票的人工核对过程控制在 5 分钟以内；
- 所有低置信度或关键信息缺失的发票均有明确提示；
- 报销批次金额与纳入统计的发票金额合计一致；
- 预览、下载和最终打印使用同一拼版逻辑；
- 老版本数据升级后不丢失发票和文件；
- 断网状态不影响任何 V6.3 功能。

## 6. 目标用户与使用场景

### 6.1 目标用户

- 经常出差并需要整理机票、火车票、住宿发票的个人；
- 每月需要汇总交通、餐饮、办公等费用的员工；
- 希望发票资料完全保存在本地的用户；
- 不愿学习复杂财务软件，只需要快速整理和打印的用户。

### 6.2 典型场景

#### 场景 A：出差报销

用户导入机票、火车票、酒店发票和行程单，创建“2026年6月北京出差”批次，核对金额和日期后生成报销单及拼版 PDF。

#### 场景 B：月度费用整理

用户将一个月内的交通、餐饮、办公发票导入软件，通过分类和状态筛选完成核对，并分别归入不同报销批次。

#### 场景 C：历史发票查找

用户通过销售方、发票号码、金额、日期或批次名称，找到过去已经报销的发票并打开原始文件。

#### 场景 D：数据迁移

用户更换电脑前导出本地备份，在新电脑安装 InvoiceBox 后恢复全部发票、设置、批次和分类学习规则。

## 7. 核心业务流程

### 7.1 主流程

1. 用户导入 PDF 或 OFD 文件；
2. 系统完成文件去重、格式转换、OCR 和自动分类；
3. 系统将低置信度、信息缺失或疑似重复的发票标记为待处理；
4. 用户进入“待确认”视图逐张或批量核对；
5. 用户创建或选择报销批次；
6. 用户将发票移动到报销批次；
7. 用户检查批次金额、异常项和报销单；
8. 用户生成报销材料；
9. 系统将批次和相关发票标记为“已导出”；
10. 用户可继续标记为“已报销”或“已归档”。

### 7.2 状态流转

发票状态：

```text
待确认 pending
   ↓
已确认 confirmed
   ↓
已导出 exported
   ↓
已报销 reimbursed
   ↓
已归档 archived
```

允许的特殊操作：

- 已确认可以退回待确认；
- 已导出可以重新导出；
- 已报销可以退回已确认；
- 已归档可以恢复为已报销；
- 删除操作独立于状态，并始终要求二次确认。

报销批次状态：

```text
草稿 draft
   ↓
已导出 exported
   ↓
已报销 reimbursed
   ↓
已归档 archived
```

## 9. 详细功能需求

## 9.1 V63-01 三栏式工作台

### 需求说明

现有界面由发票列表和预览区组成。V6.3 增加详情面板，并重新组织筛选和批次入口。

### 交互要求

- 右侧详情面板默认宽度建议为 340 至 400px；
- 用户点击发票卡片后，详情面板展示该发票；
- 当前选中发票需要有明显高亮；
- 用户点击批次后，列表仅显示该批次内的发票；
- 窗口宽度不足时，详情面板允许折叠；
- 不改变当前桌面优先的产品定位，不要求适配手机屏幕。

### 验收标准

- 发票列表、预览和详情能够同时展示；
- 切换发票后详情与预览同步更新；
- 面板滚动互不干扰；
- 1366×768 分辨率下核心操作可用。

## 9.3 V63-03 待确认与异常筛选

### 快捷筛选

- 全部发票；
- 待确认；
- 低置信度；
- 信息缺失；
- 疑似重复；
- 已确认；
- 已导出；
- 已报销；
- 已归档。

### 低置信度规则

满足以下条件之一：

- `confidence < 0.70`；
- `needs_manual_review=true`。

### 信息缺失规则

满足以下任一条件：

- 金额为空或小于等于 0；
- 分类为空或为“其他”；
- 发票日期和费用日期均为空；
- OCR 未能提取任何有效文本；
- 需要发票号码的增值税发票未提取到发票号码。

机票、火车票、行程单等票据不强制要求发票号码。

### 列表提示

- 待确认：橙色“需确认”；
- 低置信度：红色圆点或标签；
- 信息缺失：灰色“信息不全”；
- 疑似重复：紫色“疑似重复”；
- 已确认：绿色状态；
- 已报销和已归档：降低列表视觉权重。

### 验收标准

- 筛选结果数量准确；
- 一张发票可同时具有多个异常标签；
- 用户完成确认或补齐信息后，筛选结果实时更新；
- 筛选不修改发票数据。

## 9.4 V63-04 发票状态管理

### 状态字段

| 状态值 | 中文名称 | 使用说明 |
|---|---|---|
| pending | 待确认 | 新导入或需要人工检查 |
| confirmed | 已确认 | 信息已核对 |
| exported | 已导出 | 已生成报销材料 |
| reimbursed | 已报销 | 用户确认报销完成 |
| archived | 已归档 | 历史资料，默认可隐藏 |

### 默认状态

- `needs_manual_review=true`：默认 `pending`；
- `needs_manual_review=false` 且关键信息完整：默认 `confirmed`；
- 老版本数据迁移：默认根据置信度和信息完整性计算状态。

### 验收标准

- 状态可手动修改；
- 批次成功导出后，批次内 `confirmed` 发票变为 `exported`；
- 导出失败不得修改发票状态；
- 已归档发票默认不出现在“全部活动发票”中，但可通过筛选查看。

## 9.5 V63-05 报销批次管理

### 批次字段

```json
{
  "id": "uuid",
  "name": "2026年6月北京出差",
  "applicant": "",
  "department": "",
  "project": "",
  "description": "",
  "status": "draft",
  "created_at": "2026-06-22T10:00:00",
  "updated_at": "2026-06-22T10:00:00"
}
```

### 批次操作

- 新建批次；
- 编辑批次；
- 删除空批次；
- 将发票移动到批次；
- 将发票移出批次；
- 标记批次已报销；
- 归档批次；
- 恢复归档批次。

### 删除规则

- 空批次可以直接二次确认后删除；
- 非空批次删除前必须让用户选择：
  - 将发票移到“未分组”；
  - 取消删除；
- 删除批次不能删除发票文件。

### 批次详情汇总

- 发票数量；
- 纳入报销金额的发票数量；
- 总金额；
- 分类汇总；
- 待确认数量；
- 信息缺失数量；
- 疑似重复数量；
- 最近更新时间。

### 默认批次

系统提供不可删除的“未分组”视图，但不需要在数据中创建固定批次记录。`batch_id` 为空即表示未分组。

### 验收标准

- 创建、编辑和删除批次后刷新页面结果不丢失；
- 发票只能属于一个报销批次；
- 批次金额只统计应纳入报销金额的发票；
- “行程单”等排除分类不重复计入总金额；
- 删除批次不会删除任何发票文件。

## 9.6 V63-06 批量操作

### 进入方式

- 发票列表增加复选框；
- 选择一张或多张后显示批量操作栏；
- 提供“选择当前筛选结果”和“取消选择”。

### 支持操作

- 批量确认；
- 批量修改分类；
- 批量移动到报销批次；
- 批量修改状态；
- 批量删除；
- 批量移出当前批次。

### 操作规则

- 批量删除必须二次确认，并显示文件数量；
- 批量修改分类需要记录每张发票的纠错学习；
- 批量确认不能自动补齐缺失金额；
- 对信息仍不完整的发票执行批量确认时，应提示风险；
- 批量操作部分失败时，显示成功和失败数量。

### 验收标准

- 批量操作不会影响未选中的发票；
- 当前筛选变化后应清理无效选择；
- 100 张发票批量修改时界面有处理状态；
- 操作完成后列表、汇总和预览同步刷新。

- 金额固定保留两位小数；
- 日期统一使用 `YYYY-MM-DD`；
- 所有文件名清理 Windows 非法字符。

- 批次状态变为 `exported`；
- 批次中状态为 `confirmed` 的发票变为 `exported`；
- 待确认发票保持 `pending`，即使用户强制导出；
- 写入 `last_exported_at`。

- ZIP 中四个文件均可正常打开；
- 报销单、CSV 和拼版 PDF 金额一致；
- 导出失败不修改任何状态；
- 相同批次允许重复导出；
- 重复导出文件名自动增加时间或序号，不能静默覆盖。

其中 `user_classifications.json` 不存在时允许跳过。

## 9.12 V63-12 基础业务重复提示

### 判断规则

在现有 MD5 精确去重之外，增加基础业务重复提示。

1. 发票号码相同且发票号码非空；
2. 销售方、金额、开票日期均相同；
3. 销售方、金额、费用开始日期均相同。

### 处理原则

- MD5 完全重复继续拒绝导入；
- 业务疑似重复允许导入；
- 不自动删除或覆盖；
- 在列表和详情中展示疑似重复提示；
- 用户可以选择“确认不是重复”，状态改为 `ignored`；
- V6.3 不实现双票并排对比页面。

### 验收标准

- 完全相同文件仍被准确拦截；
- 疑似重复发票不会被自动删除；
- 用户忽略后不再出现在“疑似重复”筛选中；
- 发票号码为空时不能仅凭空值判断重复。

## 13. 非功能需求

### 13.1 离线与隐私

- Flask 仅监听 `127.0.0.1`；
- 禁止主动请求外部 URL；
- 禁止使用在线统计、崩溃上报和遥测；
- 禁止将发票内容发送到任何远程服务；
- 所有前端依赖继续本地打包。

### 13.2 性能

- 100 张发票列表滚动应保持可用；
- 普通筛选和搜索结果应在 300ms 内更新；
- 批量操作需要显示处理中状态；
- 大型预览生成不能重复触发并发请求；
- 切换仅影响详情信息时，不应重新生成全部拼版页面；

### 13.3 稳定性

- 关键文件操作使用 try/except；
- OCR、OFD 转换、PDF 拼版和 ZIP 恢复失败不能导致应用退出；
- JSON 写入使用 UTF-8、`ensure_ascii=False` 和原子替换；
- 数据文件损坏时保留损坏副本；
- 删除文件失败时不能先删除数据库记录。

### 13.4 安全

- 防止文件路径穿越；
- 防止 ZIP 路径穿越；
- 文件接口只允许访问 `UPLOAD_DIR` 内文件；
- 不信任前端传入的文件名、分类、金额、状态和 ID；
- 恢复包必须先验证再替换；
- 删除和恢复操作必须二次确认。

### 13.5 可用性

- 关键操作提供成功或失败反馈；
- 不允许静默吞掉用户可感知错误；
- 危险操作使用明确的中文说明；
- 空状态、加载状态、失败状态均需设计；
- 按钮文案使用业务语言，避免暴露技术术语。

## 17. 风险与处理方案

### 风险 1：V6.3 改动范围较大

处理：

- 先升级数据层，再调整界面；
- 每阶段保持现有上传、预览、打印能力可用；
- 不同时加入图片、ZIP 和目录监听。

### 风险 2：旧数据字段不完整

处理：

- 所有新字段设置默认值；
- 不强制回填无法可靠推断的信息；
- 保存时再逐步升级记录格式。

### 风险 3：金额口径不一致

处理：

- 后端建立统一的报销金额计算函数；
- 报销单、批次汇总、CSV 和异常检查共同调用；
- 前端展示以后端结果为准。

### 风险 4：恢复操作损坏数据

处理：

- 恢复前自动备份；
- 临时目录验证；
- 全部验证成功后再替换；
- 任一步失败都保持原数据。

### 风险 5：拼版预览性能下降

处理：

- 编辑普通字段时不重新渲染全部页面；
- 只有分类、顺序、批次选择或打印设置变化时刷新拼版；
- 防止短时间内重复生成。
